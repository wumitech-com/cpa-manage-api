# 自动养号脚本监控方案设计

## 📋 需求分析

你的Shell脚本核心功能：
1. ✅ 检查Python脚本日志文件是否持续更新
2. ✅ 根据文件修改时间判断脚本运行状态
3. ✅ 检查进程和Screen会话状态
4. ✅ 自动重启停止的脚本
5. ✅ 记录和展示监控信息

**目标：** 将这套监控逻辑集成到Java后端，实现可视化、可管理、可扩展的监控系统

---

## 🎯 方案对比

### 方案1：简单定时任务 ⭐

**适用场景：** 单台服务器，简单监控，快速上线

#### 技术方案
```
Spring Boot 定时任务 (@Scheduled)
    ↓
通过SSH执行Shell命令检查日志
    ↓
判断状态并自动重启
    ↓
日志记录到logback
```

#### 优点
- ✅ 实现简单，代码量少（~100行）
- ✅ 无需数据库表，不改变现有结构
- ✅ 快速上线（1-2小时）
- ✅ 适合单设备或少量设备

#### 缺点
- ❌ 无历史记录，无法追溯
- ❌ 无可视化界面
- ❌ 难以扩展到多设备
- ❌ 状态无法持久化

#### 实现要点
```java
@Service
public class SimpleMonitorService {
    @Scheduled(fixedRate = 300000) // 5分钟
    public void check() {
        // 1. SSH执行: stat命令获取文件修改时间
        // 2. 计算时间差
        // 3. 判断是否需要重启
        // 4. SSH执行: screen命令重启脚本
    }
}
```

#### 成本评估
- **开发时间：** 0.5天
- **维护成本：** 低
- **扩展性：** ⭐

#### 数据库表设计
```sql
script_monitor_log (监控日志表)
├── id (主键)
├── device_id (设备标识)
├── script_name (脚本名称)
├── status (状态: RUNNING/PAUSED/STOPPED/ERROR)
├── time_diff (距上次输出的秒数)
├── log_file_path (日志文件路径)
├── log_file_size (文件大小)
├── has_process (是否有进程)
├── has_screen (是否有Screen会话)
├── restarted (是否触发重启)
├── recent_logs (最近10行日志)
├── check_time (检查时间)
└── remark (备注)
```

#### 优点
- ✅ 完整的历史记录，可追溯
- ✅ 支持多设备监控
- ✅ 提供API接口，易于集成前端
- ✅ 可查询统计（重启次数、故障时长等）
- ✅ 支持手动检查和重启
- ✅ 易于扩展和维护

#### 缺点
- ❌ 需要新增数据库表
- ❌ 代码量相对较多（~500行）
- ❌ 数据量会持续增长，需定期清理

#### API功能清单
```
1. 获取所有设备最新状态
   GET /api/monitor/script/latest
   返回：所有设备的最新监控状态

2. 手动检查指定设备
   POST /api/monitor/script/check/10.7.136.129
   返回：立即执行检查并返回结果

3. 手动重启脚本
   POST /api/monitor/script/restart/10.7.136.129

4. 查询历史记录
   GET /api/monitor/script/history/10.7.136.129?startTime=xxx&endTime=xxx
   返回：指定时间范围的监控记录

5. 查询今日历史
   GET /api/monitor/script/history/10.7.136.129/today
   返回：当天所有监控记录
```

## 📊 方案2详细设计（推荐实现）

### 核心流程

```
1. 定时任务启动（每5分钟）
   ↓
2. 遍历配置的设备列表
   ↓
3. 对每台设备执行检查
   ├─ SSH连接设备
   ├─ 执行stat命令获取日志文件修改时间
   ├─ 计算时间差
   ├─ 检查进程状态 (ps aux)
   ├─ 检查Screen会话 (screen -list)
   ├─ 获取最近10行日志 (tail)
   └─ 判断状态
   ↓
4. 根据状态决定是否重启
   ├─ 时间差 > 30分钟 → 重启
   ├─ 日志文件不存在 → 重启
   └─ 其他 → 记录状态
   ↓
5. 保存监控记录到数据库
   ↓
6. 等待下次检查
```

### 监控配置结构

```java
class MonitorConfig {
    String deviceId;      // "10.7.136.129"
    String logDir;        // "/data/appium/.../log"
    String scriptName;    // "TTTest_main_auto_1"
    String scriptPath;    // "/data/appium/.../zl"
    String screenName;    // "10.7.136.129-tt_main_auto-001"
}
```

### SSH命令封装

```bash
# 1. 检查日志文件状态
if [ -f {logFile} ]; then
  MTIME=$(stat -c%Y {logFile});
  NOW=$(date +%s);
  DIFF=$((NOW - MTIME));
  SIZE=$(stat -c%s {logFile});
  echo $DIFF;
  echo $SIZE;
else
  echo -1;
  echo 0;
fi

# 2. 检查进程
ps aux | grep {scriptName} | grep -v grep | wc -l

# 3. 检查Screen
screen -list | grep {screenName} | wc -l

# 4. 获取最近日志
tail -n 10 {logFile}

# 5. 重启脚本
mkdir -p {logDir};
screen -S {screenName} -X quit 2>/dev/null || true;
screen -S {screenName} -X quit 2>/dev/null || true;
sleep 1;
screen -S {screenName} -dm;
screen -S {screenName} -X stuff 'cd {scriptPath} && python3 {scriptName}.py\n'
```

### 状态判断逻辑

```
时间差 == -1 → STOPPED (日志文件不存在)
时间差 > 1800秒 → STOPPED (30分钟没更新)
时间差 > 300秒 → PAUSED (5分钟没更新)
时间差 <= 300秒 → RUNNING (正常运行)
```

### 自动重启条件

```
满足以下任一条件触发重启：
1. status == STOPPED
2. 日志文件不存在
3. 连续3次检查都是PAUSED状态（可选）
```

## 💡 关键技术点

### 1. SSH连接复用

```java
// 使用连接池，避免频繁建立连接
// 设置合理的超时时间
SshUtil.executeCommand(deviceId, command, timeout=30s);
```

### 2. 并发处理

```java
// 使用线程池并发检查多台设备
ExecutorService executor = Executors.newFixedThreadPool(5);
for (MonitorConfig config : configs) {
    executor.submit(() -> checkSingleScript(config));
}
```

### 3. 异常处理

```java
// SSH连接失败、命令执行失败都要有兜底方案
try {
    // 检查逻辑
} catch (SshException e) {
    log.error("SSH连接失败，设备可能离线");
    saveErrorStatus("SSH_ERROR");
} catch (Exception e) {
    log.error("检查异常", e);
    saveErrorStatus("CHECK_ERROR");
}
```

### 4. 防止重复重启

```java
// 检查最近5分钟是否已经重启过
LocalDateTime recentTime = LocalDateTime.now().minusMinutes(5);
if (hasRecentRestart(deviceId, recentTime)) {
    log.warn("设备{}最近已重启过，跳过本次重启", deviceId);
    return;
}
```

### 5. 数据清理策略

```sql
-- 定期清理30天前的监控记录
DELETE FROM script_monitor_log 
WHERE check_time < DATE_SUB(NOW(), INTERVAL 30 DAY);

-- 或者只保留最新的1000条记录
DELETE FROM script_monitor_log 
WHERE id NOT IN (
  SELECT id FROM (
    SELECT id FROM script_monitor_log 
    WHERE device_id = '10.7.136.129'
    ORDER BY check_time DESC LIMIT 1000
  ) tmp
);
```
