# K8s 部署对 SSH 连接的影响分析

## 当前应用 SSH 连接架构

应用需要：
1. **通过跳板机连接**：106.75.152.136
2. **目标服务器**：10.7.107.224
3. **使用私钥认证**：SSH 密钥配置在 application-prod.yml 中

## 部署到 K8s 后的影响分析

### ✅ 不会有影响的情况

1. **网络连通性**
   - Pod 可以访问外网（跳板机 106.75.152.136 是公网 IP）
   - Pod 可以访问内网（目标服务器 10.7.107.224 是内网 IP）
   - 只要 K8s 集群网络策略允许出站连接，SSH 连接应该正常

2. **SSH 配置**
   - SSH 密钥通过配置文件注入，不受部署方式影响
   - 应用代码逻辑不变，SSH 连接功能不受影响

3. **功能完整性**
   - 所有业务功能保持不变
   - SSH 连接池机制正常工作

### ⚠️ 需要注意的事项

1. **网络策略（NetworkPolicy）**
   - 如果集群启用了 NetworkPolicy，需要确保允许 Pod 访问：
     - 跳板机：106.75.152.136:22
     - 目标服务器：10.7.107.224:22（通过跳板机）

2. **DNS 解析**
   - 确保 Pod 可以解析跳板机和目标服务器的域名（如果有）
   - 通常使用 IP 地址不会有问题

3. **安全组/防火墙**
   - 确保 K8s 节点可以访问跳板机 106.75.152.136:22
   - 确保跳板机可以访问目标服务器 10.7.107.224:22

4. **资源限制**
   - 当前配置：内存 512Mi-2Gi，CPU 500m-2000m
   - SSH 连接会占用一些内存和 CPU，当前配置应该足够

## 建议的验证步骤

部署后验证：

1. **检查 Pod 网络连通性**
   ```bash
   kubectl exec -n wumitech-cpa <pod-name> -- ping -c 3 106.75.152.136
   ```

2. **测试 SSH 连接**
   ```bash
   kubectl exec -n wumitech-cpa <pod-name> -- sh -c "echo 'SSH test'"
   # 查看应用日志，确认 SSH 连接是否成功
   ```

3. **查看应用日志**
   ```bash
   kubectl logs -f -n wumitech-cpa -l app=cpa-manage-api | grep -i ssh
   ```

## 结论

**部署到 K8s 对 SSH 连接和功能应该没有影响**，前提是：
- ✅ K8s 集群网络允许出站连接
- ✅ 没有过于严格的 NetworkPolicy 限制
- ✅ 安全组/防火墙规则允许访问

如果遇到 SSH 连接问题，主要检查：
1. Pod 网络连通性
2. 网络策略配置
3. 安全组/防火墙规则
4. 应用日志中的 SSH 错误信息
