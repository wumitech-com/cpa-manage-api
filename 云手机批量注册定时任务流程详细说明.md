# 云手机批量注册账号定时任务流程详细说明

## 📋 概述

云手机批量注册账号定时任务是一个基于Spring Boot `@Scheduled`注解的定时任务系统，每60秒执行一次，自动查询并执行待注册的云手机任务。

---

## 🔄 整体流程架构

```
定时任务调度器 (每60秒)
    ↓
查询PENDING状态的云手机任务
    ↓
并发控制 (最多7个并发)
    ↓
设备锁机制 (防止同一设备被多个任务同时操作)
    ↓
更新任务状态为RUNNING
    ↓
异步执行注册流程
    ↓
单设备注册流程:
  1. ResetPhoneEnv (重置环境+换机)
  2. 安装TikTok APK
  3. 添加Hook配置
  4. 执行注册脚本
  5. 解析并保存账号信息
  6. 获取流量数据
  7. 调用备份接口
    ↓
更新任务状态 (COMPLETED/FAILED/STOPPED)
```

---

## 📝 详细流程说明

### 1. 定时任务入口

**位置**: `TtRegisterService.scheduledExecutePendingTasks()`

**执行频率**: 每60秒执行一次 (`@Scheduled(fixedRate = 60 * 1000)`)

**主要逻辑**:
```java
1. 查询数据库，获取所有PENDING状态的云手机任务
   - 条件: status = 'PENDING'
   - 条件: device_type IS NULL OR device_type = 'CLOUD_PHONE'
   - 排序: 按created_at降序

2. 如果没有待执行任务，直接返回

3. 如果有待执行任务，开始并发执行
   - 最大并发数: 7 (使用Semaphore控制)
   - 每个任务异步执行
```

---

### 2. 任务状态检查与并发控制

**双重检查机制**:
```java
1. 内存检查: 检查stoppedTaskIds集合（快速响应停止请求）
2. 数据库检查: 再次查询数据库确认任务状态
```

**设备锁机制**:
```java
- 使用ConcurrentHashMap存储每个设备的Semaphore
- 每个设备同一时间只能被一个任务操作
- 如果设备正在被使用，跳过该任务
```

**并发控制**:
```java
- 使用Semaphore控制最大并发数为7
- 如果并发数已达上限，跳过该任务（下次定时任务会再次尝试）
```

---

### 3. 任务执行流程

**位置**: `TtRegisterService.executeTaskForDevice()`

#### 3.1 任务类型判断

根据`targetCount`判断任务类型：

**无限循环注册** (`targetCount == 0`):
```java
while (true) {
    // 检查停止信号
    // 执行注册流程
    // 更新心跳时间
    // 休息5秒后继续下一轮
}
```

**指定数量注册** (`targetCount > 0`):
```java
for (int i = 1; i <= targetCount; i++) {
    // 检查停止信号
    // 执行注册流程
    // 更新心跳时间
    // 休息5秒后继续下一个账号
}
```

#### 3.2 单设备注册流程

**位置**: `TtRegisterService.registerSingleDeviceWithoutStart()`

**完整步骤**:

##### 步骤1: 准备ResetPhoneEnv参数

```java
1. 提取国家代码 (country)
   - 从resetParams获取
   - 如果为空，从phoneId中提取（如tt_107_224_1 -> US）

2. 提取SDK版本 (sdk)
   - 从resetParams获取
   - 默认值: "33"

3. 获取镜像路径 (imagePath)
   - 从resetParams获取
   - 如果为空，通过docker inspect获取设备镜像名称

4. 提取GAID标签 (gaidTag)
   - 从resetParams获取
   - 如果为空，从phoneId中提取

5. 随机选择动态IP渠道 (dynamicIpChannel)
   - 从 ["ipidea", "closeli", "ipbiubiu"] 中随机选择
   - 每次调用都重新随机选择

6. 获取静态IP渠道 (staticIpChannel)
   - 从resetParams获取
   - 默认值: 配置的默认静态IP渠道

7. 获取业务参数 (biz)
   - 从resetParams获取
```

##### 步骤2: 调用ResetPhoneEnv接口

**重试机制**:
- 最多重试5次
- 首次调用前随机延迟0-3秒（避免并发冲突）
- 重试时等待3秒（网关错误使用指数退避，最多30秒）

**并发控制**:
- 按服务器IP分组，每个服务器最多同时7个ResetPhoneEnv调用
- 使用Semaphore控制并发数

**接口调用**:
```java
apiService.resetPhoneEnv(
    phoneId,           // 设备ID
    serverIp,          // 服务器IP
    country,           // 国家代码
    sdk,              // SDK版本
    imagePath,        // 镜像路径
    gaidTag,          // GAID标签
    dynamicIpChannel, // 动态IP渠道
    staticIpChannel,  // 静态IP渠道
    biz               // 业务参数
)
```

**返回结果处理**:
- 从响应中提取`realIp`（真实IP地址）
- 从响应中提取`gaid`（Google Advertising ID）
- 等待10秒，确保reset和换机完成

##### 步骤3: 安装TikTok APK

**位置**: `TtRegisterService.installTikTokApk()`

**两种安装方式**:

**方式1: 使用install.sh脚本** (如果存在)
```bash
1. 复制APK目录到容器: docker cp /apk_repo/{tiktokVersionDir} {phoneId}:/
2. 在容器内执行安装脚本: docker exec {phoneId} sh -c 'cd /{tiktokVersionDir} && sh install.sh {phoneId}'
```

**方式2: 直接安装APK文件** (如果不存在install.sh)
```bash
1. 查找APK文件: ls /apk_repo/{tiktokVersionDir}/*.apk
2. 复制APK到容器: docker cp {apkPath} {phoneId}:/{apkFileName}
3. 安装APK: docker exec {phoneId} sh -c "pm install -r /{apkFileName}"
```

**验证安装**:
- 检查脚本输出中是否包含"install success"或"success"
- 验证应用包是否已安装: `docker exec {phoneId} adb shell pm list packages | grep com.zhiliaoapp.musically`

**等待时间**: 安装成功后等待5秒

##### 步骤4: 添加Hook配置

**位置**: `TtRegisterService.addAppHookConfig()`

**执行步骤**:
```bash
1. 复制hook脚本到容器:
   docker cp /home/ubuntu/agent/scripts/add_app_hook_config.sh {phoneId}:/system/etc/add_app_hook_config.sh

2. 添加执行权限:
   docker exec {phoneId} chmod +x /system/etc/add_app_hook_config.sh

3. 执行hook脚本:
   docker exec {phoneId} sh /system/etc/add_app_hook_config.sh com.zhiliaoapp.musically
```

**注意**: Hook配置失败不会中断流程，只记录警告

##### 步骤5: 处理TikTok版本

**位置**: `TtRegisterService.extractTiktokVersion()`

**版本提取逻辑**:
- 从`tiktokVersionDir`中提取版本号
- 支持格式: `com.zhiliaoapp.musically_us_43.1.4` -> `43.1.4`
- 如果无法提取，返回失败

##### 步骤6: 执行注册脚本

**位置**: `TtRegisterService.executeRegisterScript()`

**脚本信息**:
- **服务器**: `10.13.55.85` (Appium服务器)
- **用户**: `ubuntu`
- **脚本路径**: `/data/appium/com_zhiliaoapp_musically/tiktok_register_us_test_account.py`
- **超时时间**: 20分钟

**脚本参数** (8个参数):
```bash
1. offerid: "test"
2. phoneId: 设备ID
3. phoneServerIp: 云手机服务器IP（直接使用serverIp，不需要IP映射）
4. appiumServer: 随机选择的Appium服务器端口
5. country: 国家代码
6. emailMode: "random"（假邮箱）或 "outlook"（真邮箱）
7. adbPort: ADB端口（云手机为空字符串）
```

**执行命令**:
```bash
cd /data/appium/com_zhiliaoapp_musically && \
timeout 20m stdbuf -o0 -e0 python3 -u \
  tiktok_register_us_test_account.py \
  {offerid} {phoneId} {phoneServerIp} {appiumServer} {country} {emailMode} {adbPort}
```

**执行结果判断**:
- **成功标识**: 
  - 退出码为0
  - 输出中包含"注册成功！账号信息："或"end do register ."
- **失败标识**:
  - 退出码为124（超时）
  - 退出码为1（Python环境问题）
  - 输出中包含"failed and exit(2) from chen"

**进程检查**:
- 如果退出码为0但进程仍在运行，认为是异常情况
- 如果超时，尝试清理僵尸进程

##### 步骤7: 解析并保存账号信息

**位置**: `TtRegisterService.parseRegisterOutput()`

**解析字段**:
```java
- email: 邮箱账号
- password: 密码
- username: TikTok用户名
- behavior: 行为结果（nickname_behavior_result）
- gaid: Google Advertising ID（从ResetPhoneEnv获取）
- androidVersion: Android版本（从SDK转换）
- ipChannel: IP渠道（优先使用dynamicIpChannel）
- ip: IP地址（从ResetPhoneEnv的realIp获取）
- nurtureJsonSource: "ResetPhoneEnv"（固定值）
```

**保存到数据库**:
- 表名: `tt_account_register`
- 设置`registerSuccess = true`
- 设置`registerType = "FAKE_EMAIL"`或`"REAL_EMAIL"`（根据emailMode）

##### 步骤8: 获取流量数据

**位置**: `TtRegisterService.executeRegisterScript()` (注册成功后)

**API调用**:
```java
String trafficData = apiService.getTrafficData(phoneId);
```

**更新数据库**:
- 将流量数据保存到`tt_account_register.traffic_data`字段
- 如果获取失败，不影响注册成功状态

##### 步骤9: 调用备份接口

**调用条件**:

**假邮箱** (`FAKE_EMAIL`):
- `is_2fa_setup_success = true` **且** `registerSuccess = true`

**真邮箱** (`REAL_EMAIL`):
- `registerSuccess = true`

**API调用**:
```java
boolean backupResult = apiService.backupApp(
    phoneId, 
    serverIp, 
    "com.zhiliaoapp.musically"
);
```

**保存结果**:
- 将备份结果保存到`tt_account_register.backup_success`字段
- 如果备份失败，不影响注册成功状态

---

### 4. 任务状态更新

**任务状态流转**:
```
PENDING → RUNNING → COMPLETED/FAILED/STOPPED
```

**状态说明**:
- **PENDING**: 待执行（定时任务会查询并执行）
- **RUNNING**: 执行中（定时任务不会重复执行）
- **COMPLETED**: 已完成（所有账号注册完成）
- **FAILED**: 失败（执行过程中发生异常）
- **STOPPED**: 已停止（用户手动停止或检测到停止信号）

**心跳更新**:
- 每个账号注册完成后，更新`updated_at`字段
- 用于检测长时间未更新的任务（卡住的任务）

---

### 5. 卡住任务检测

**位置**: `TtRegisterService.scheduledCheckStuckTasks()`

**执行频率**: 每5分钟执行一次 (`@Scheduled(fixedRate = 5 * 60 * 1000)`)

**检测逻辑**:
```java
1. 查询超过1小时未更新的RUNNING状态任务
2. 双重检查：再次查询数据库确认状态
3. 如果状态仍为RUNNING，重置为PENDING
4. 重置后，下次定时任务会重新执行
```

**注意**: 不会重置STOPPED状态的任务

## 🔧 关键配置

### 并发控制配置

```java
// 定时任务最大并发数
int maxConcurrency = 7;

// ResetPhoneEnv接口每个服务器最大并发数
int MAX_RESET_PHONE_ENV_CONCURRENCY_PER_SERVER = 7;
```

### 超时配置

```java
// 注册脚本超时时间
int scriptTimeoutMinutes = 20; // 20分钟

// 卡住任务检测时间阈值
LocalDateTime oneHourAgo = LocalDateTime.now().minusHours(1); // 1小时
```

### 重试配置

```java
// ResetPhoneEnv接口最大重试次数
int maxRetries = 5;

// 重试等待时间
- 首次调用: 随机延迟0-3秒
- 普通错误: 3秒
- 超时错误: 5秒
- 网关错误: 5秒 + 指数退避（最多30秒）
```

## 📊 数据库表结构

### tt_register_task (任务表)

**关键字段**:
- `task_id`: 任务ID（唯一标识）
- `phone_id`: 云手机ID
- `server_ip`: 云手机服务器IP
- `status`: 任务状态（PENDING/RUNNING/COMPLETED/FAILED/STOPPED）
- `target_count`: 目标注册数量（0表示无限循环）
- `task_type`: 任务类型（FAKE_EMAIL/REAL_EMAIL）
- `device_type`: 设备类型（CLOUD_PHONE/MAINBOARD）
- `created_at`: 创建时间
- `updated_at`: 更新时间（心跳时间）

### tt_account_register (账号注册表)

**关键字段**:
- `phone_id`: 云手机ID
- `phone_server_ip`: 云手机服务器IP
- `email`: 邮箱账号
- `password`: 密码
- `username`: TikTok用户名
- `register_success`: 注册是否成功
- `register_type`: 注册类型（FAKE_EMAIL/REAL_EMAIL）
- `backup_success`: 备份是否成功
- `traffic_data`: 流量数据
- `nurture_json_source`: 养号json来源（固定为"ResetPhoneEnv"）

## 🚨 异常处理

### 1. ResetPhoneEnv接口异常

**处理方式**:
- 最多重试5次
- 根据错误类型使用不同的等待时间
- 网关错误使用指数退避策略

### 2. 注册脚本超时

**处理方式**:
- 20分钟超时后自动终止
- 尝试清理僵尸进程
- 如果是真邮箱，尝试解析已获取的Outlook邮箱信息并保存

### 3. 脚本进程卡住

**处理方式**:
- 检查脚本进程是否仍在运行
- 如果SSH通道关闭但进程仍在运行，强制终止进程

### 4. 任务被停止

**处理方式**:
- 内存集合快速检查（`stoppedTaskIds`）
- 数据库双重检查
- 检测到停止信号后立即退出循环

## 📈 性能优化

### 1. 并发控制

- 定时任务级别：最多7个并发
- ResetPhoneEnv级别：每个服务器最多7个并发
- 设备级别：每个设备同一时间只能被一个任务操作

### 2. 随机延迟

- ResetPhoneEnv首次调用前随机延迟0-3秒，避免并发冲突
- 动态IP渠道随机选择，分散负载

### 3. 异步执行

- 所有任务异步执行，不阻塞定时任务
- 使用`CompletableFuture`管理异步任务

## 🎯 总结

云手机批量注册定时任务系统通过以下机制实现了稳定、高效的批量注册：

1. **定时调度**: 每60秒自动检查并执行待注册任务
2. **并发控制**: 多层次的并发控制，避免资源竞争
3. **设备锁**: 防止同一设备被多个任务同时操作
4. **重试机制**: ResetPhoneEnv接口支持重试，提高成功率
5. **异常处理**: 完善的异常处理和日志记录
6. **卡住检测**: 自动检测并恢复卡住的任务
7. **状态管理**: 清晰的任务状态流转和心跳更新机制

整个系统设计合理，能够稳定运行并处理各种异常情况。
